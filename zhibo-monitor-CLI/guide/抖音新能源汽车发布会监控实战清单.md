# 抖音新能源汽车发布会监控实战清单

## 1. 目标

今天的目标不是一次性做全量分析，而是先稳定跑通一场抖音新能源汽车发布会的基础监控链路：

- 成功进入直播间
- 持续抓取直播消息
- 持续记录在线人数和点赞展示值
- 能在直播结束后拿到原始数据库结果

当前项目已经具备抖音 MVP 采集能力，适合先完成以下 P0 指标：

- 直播热度表现
  - 在线人数展示值峰值
  - 在线人数展示值均值
  - 点赞展示值
- 互动表现
  - 消息总量
  - 发言人数
  - 消息密度
- 内容留存
  - 原始消息文本留存，供后续情感分析和关键词分析

说明：

- 当前抖音采集器抓到的 `online_count` 和 `like_count` 都更接近页面展示值，不一定等于平台真实在线人数或真实累计点赞。
- 当前采集方式以 DOM 解析为主，先以跑通真实发布会为第一目标。

## 2. 开播前准备

### 2.1 环境确认

在项目根目录执行：

```bash
./venv/bin/python main.py --help
./venv/bin/python main.py init-db
```

当前项目已支持在 SQLite 路径父目录不存在时自动创建目录，因此默认数据库初始化应可直接成功。

### 2.2 确认房间号和直播链接

你提供的直播链接为：

```text
https://live.douyin.com/255528271839?from_search=true&is_aweme_tied=1&search_id=20260623184912F260CCE918E73A99D8FE&search_result_id=7654374649718902042
```

其中真正需要的核心信息是：

- `room_id`: `255528271839`
- 标准直播地址：`https://live.douyin.com/255528271839`

建议启动命令里直接使用 `room_id`，不要依赖带搜索参数的长链接。

### 2.3 明确任务元信息

建议在启动前先确定两项元信息，避免后续数据混乱：

- `event_name`
  - 例如：`2026 某品牌新车发布会`
- `car_brand`
  - 例如：`xiaomi`、`byd`、`nio`、`xpeng`

建议统一使用英文或固定拼音，方便后续筛选。

## 4. 开播中巡检要点

### 4.1 终端应看到的正常日志

正常情况下，终端会出现类似输出：

```text
[douyin] 已打开抖音直播间: 255528271839
[douyin] 抓取到 12 条新消息
[douyin] 成功写入 16 条消息，在线人数: 356000，点赞数: 1820000
```

如果持续能看到这三类信息，说明今天的基础监控链路是通的。

### 4.2 人工巡检频率

建议至少每 5 到 10 分钟检查一次：

- 浏览器页面是否还停留在直播间
- 页面是否出现登录、验证或风控提示
- 终端是否仍有新增消息日志
- 在线人数或点赞展示值是否持续刷新

### 4.3 建议人工记录的关键时间点

今天即使不做自动事件识别，也建议人工记下几个时间点，方便赛后复盘：

- 开播时间
- 核心车型亮相时间
- 价格公布时间
- 权益公布时间
- 智驾或续航重点讲解时间
- 结束时间

这些时间点后续可以和消息密度、在线人数波动做对应分析。

## 5. 直播结束后

### 5.1 正常停止

在终端按 `Ctrl+C` 结束采集。

当前实现会在停止时尝试关闭浏览器并结束任务状态；如果缓冲区里还有未写入消息，也会在退出前做一次落库。

### 5.2 快速检查数据库里是否有数据

可以执行下面的 Python 片段做最小验证：

```bash
./venv/bin/python - <<'PY'
from app.core.database import SessionLocal
from app.models.schema import EventTask, DanmakuRecord, LiveMetric

db = SessionLocal()
try:
    print("tasks =", db.query(EventTask).count())
    print("danmakus =", db.query(DanmakuRecord).count())
    print("metrics =", db.query(LiveMetric).count())
finally:
    db.close()
PY
```

如果这三项都大于 `0`，说明今天的基础采集结果已经成功落盘。

### 5.3 快速导出赛后统计

如果你想在发布会结束后立刻看基础结果，可以直接执行：

```bash
./venv/bin/python scripts/post_event_summary.py
```

默认会读取最近一场任务，并输出：

- 消息总量
- 独立发言用户数
- 热度峰值
- 峰值出现时间
- 平均热度
- 热度采样时间范围

如果要指定某一场任务，可以使用以下方式：

```bash
./venv/bin/python scripts/post_event_summary.py --task-id 3
./venv/bin/python scripts/post_event_summary.py --event-name "2026 某品牌新车发布会"
./venv/bin/python scripts/post_event_summary.py --room-id 255528271839
```

如果后续要接自动化流程或导出到别的脚本里，可以使用 JSON 输出：

```bash
./venv/bin/python scripts/post_event_summary.py --json
```

## 7. 当前已知限制

今天这版抖音采集器属于 MVP，已知限制如下：

- 采集方式是 DOM 解析，不是 WebSocket 原始消息解析
- `online_count` 和 `like_count` 采的是页面展示值，不能直接当绝对真实人数或点赞使用
- 页面 DOM 结构如果变更，选择器可能失效
- 部分直播间可能依赖登录态，未登录时页面信息可能不完整
- 如果触发平台风控、验证或页面卡死，采集可能中断

因此今天的目标应定义为：

- 跑通一场真实发布会监控
- 沉淀可复盘的原始消息数据
- 确认抖音作为重要平台是否可稳定落地

## 8. 异常排查

### 8.1 数据库初始化失败

先执行：

```bash
./venv/bin/python main.py init-db
```

如果仍失败，检查环境变量 `DATABASE_URL` 是否被外部覆盖。

### 8.2 能打开页面但没有消息

优先检查：

- 当前房间是否真正开播
- 页面是否出现登录或验证提示
- 抖音页面结构是否有变化
- 聊天区是否在当前登录态下可见

建议改用 `--headed` 模式肉眼观察页面。

### 8.3 有消息日志但在线人数或点赞始终为 0

这通常说明页面指标选择器没有命中，需要重新检查抖音页面上的在线人数或点赞展示节点。

### 8.4 程序中途中断

优先保留终端报错信息，不要直接重开多次。先确认是：

- 网络问题
- 页面结构问题
- 浏览器被风控或页面卡死
