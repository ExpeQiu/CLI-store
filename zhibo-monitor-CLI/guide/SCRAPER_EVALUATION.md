# 数据采集维度与采集方式评估

## 1. 采集数据维度定义

为构建发布会的“评价体系”，我们需要采集以下几个核心维度的数据：

| 数据类别 | 具体字段 | 说明与用途 |
| --- | --- | --- |
| **基础元数据** | `room_id`, `platform`, `anchor_name` | 标识当前正在监控的直播间。 |
| **实时热度** | `online_count` (当前在线), `like_count` (累计点赞) | 每隔 10~30 秒采集一次，用于绘制发布会全程的热度曲线。 |
| **弹幕明细** | `msg_id`, `timestamp`, `user_name`, `content` | 核心数据。用于后续的情感分析、高频词云以及竞品提及率计算。 |
| **互动事件(可选)**| `gift_name`, `gift_value` | 打赏或特殊礼物，用于计算粉丝黏性和核心受众活跃度。 |

## 2. 不同平台采集方式评估与推荐

针对这几个主流的新能源发布会直播平台，我们从**技术可行性**、**反爬难度**和**开发成本**三个维度进行评估。

### 2.1 B站 (Bilibili)
* **协议特征**: B站直播间的 WebSocket 协议相对开放，有很多开源的实现（解包其自定义的报文头部）。弹幕在 DOM 树中也比较清晰。
* **反爬难度**: **低**。无需登录也可看到弹幕。
* **推荐采集方式**: 
  - **方案 A**: 直接使用开源的 `bilibili-api-python` 库连接其 WebSocket 房间。
  - **方案 B**: 使用 `Playwright` 监听并拦截 `wss://broadcastlv.chat.bilibili.com` 的数据帧。
* **结论**: B站最适合作为我们系统的 **第一个 MVP 平台**，开发最快，协议最稳定。

### 2.2 抖音 (Douyin)
* **协议特征**: 抖音 Web 端直播间通过 WSS 传输，采用 `protobuf` 序列化，且包含签名校验（如 `X-Bogus`、`_signature` 等）。
* **反爬难度**: **高**。容易触发风控弹窗（需滑块验证），且偶尔会强制要求登录。
* **推荐采集方式**:
  - **方案 A (纯协议)**: 维护成本极高，不推荐。
  - **方案 B (Playwright + DOM 解析)**: 通过 Playwright 打开页面，定时（如每 500ms）去提取 `.webcast-chatroom___item` 这样的 DOM 节点。虽然耗费一定 CPU，但 100% 绕过签名校验。
  - **方案 C (Playwright 拦截解密)**: 在页面注入 JS，Hook 原生的 `WebSocket.onmessage`，在前端解密后直接把明文抛给我们的 Python 脚本。
* **结论**: 抖音是最大的流量池，必须做。建议采用 **方案 B 或 C**。

### 2.3 视频号 (WeChat Channels)
* **协议特征**: PC 端视频号（网页版）近期才逐步完善，加密极其复杂，依赖微信的 Wasm 模块解密。
* **反爬难度**: **极高**。强绑定微信扫码登录，且对频繁刷新有风控。
* **推荐采集方式**:
  - 只能采用 **Playwright + DOM 解析**。或者使用类似 `mitmproxy` 的中间人代理，但这需要处理证书和复杂的加解密。
* **结论**: 建议放在第二阶段（Phase 2）再进行攻坚。

### 2.4 微博直播 (Weibo)
* **协议特征**: 微博直播融合了一直播体系，采用 WebSocket 下发数据，部分场景降级为长轮询。
* **反爬难度**: **中等偏高**。强依赖登录态，高热度直播间对游客限制严格；直接请求接口面临复杂的参数签名校验和账号风控。
* **推荐采集方式**:
  - **方案 A (Playwright + DOM 解析)**: 通过 Playwright 打开直播间，复用扫码或密码登录状态，定时提取弹幕和热度 DOM 节点。能完美绕过签名和风控。
  - **方案 B (Playwright 拦截解包)**: 拦截 WebSocket 报文进行 JSON 解析，数据更全但维护成本较高。
* **结论**: 采用 **方案 A** 落地，复用视频号的 DOM 抓取思路，通过统一的登录命令保存凭证。

## 3. 结论：第一阶段的采集策略

综上评估，第一阶段的采集策略如下：
1. **统一采集底座**：采用 `Playwright` 作为统一的底层。因为它既能做 DOM 解析，又能做网络请求拦截，且可以方便地处理后续可能出现的“登录弹窗”、“滑块验证”。
2. **首发平台：B站**。先用 B站 跑通“采集 -> 入库”的整个管道。
3. **进阶平台：抖音**。通过 Playwright 的 DOM 解析或 JS Hook 方式攻克抖音的弹幕和在线人数。
